# Anchor — Explained Like You Built It (Part 1: The Problem and The Core Engine)

## Why Anchor Exists

Every bank, every defense contractor, every insurance company that wants to use AI agents has the same unsolved problem: how do you prove to a regulator that your AI never leaked customer data, never made an unauthorized decision, and never executed code it shouldn't have? Today, the answer is "trust us, we have internal policies." That answer is not good enough when the RBI walks in and asks for evidence, or when the SEC wants to know why your credit model denied a loan.

Anchor solves this by creating a deterministic, cryptographically verifiable governance layer that wraps around any AI system. It does not replace your AI. It does not slow down your AI. It sits between your AI and the outside world and does two things: it enforces compliance rules, and it creates a tamper-proof mathematical record of every single decision the AI makes. If a regulator asks "did your AI ever leak a social security number?", you do not say "we don't think so." You hand them a signed, hash-chained audit ledger and say "here is the cryptographic proof that it did not."

The system is built as two separate codebases. The first is a Python package called anchor-audit that developers install with pip. This is the governance engine itself. The second is a web platform called the Sovereign Identity Mesh that provides dashboards, authentication, and a federated relay system for organizations to manage their governance at scale. The Python package is the brain. The web platform is the nervous system that connects multiple brains together.

## The Constitution: How Rules Are Defined

At the heart of Anchor is a file called constitution.anchor. This is a YAML manifest that declares every governance rule the system enforces. Think of it as the legal constitution of your AI system. It is not a configuration file that developers edit casually. It is a cryptographically sealed document, and if anyone tampers with it, the system refuses to run.

The constitution organizes rules into three tiers. The first tier is nine core governance domains. These are the foundational categories: security, ethics, privacy, alignment, agentic AI safety, legal compliance, operational resilience, supply chain integrity, and a shared domain for cross-cutting rules. Each domain is its own file. The security domain, for example, contains rules about preventing shell injection, blocking credential exposure, detecting prompt injection attacks, and prohibiting direct subprocess execution. The ethics domain contains rules about prohibited proxy variables in lending decisions, explainability requirements, and bias detection. The privacy domain covers PII leakage, data sovereignty, and consent tracking.

The second tier is industry frameworks. Anchor ships with mappings to the FINOS AI Governance Framework, the OWASP LLM Top 10 for 2025, and the NIST AI Risk Management Framework. These are not separate rule sets. They are translation layers. A FINOS rule like FINOS-014 maps directly to Anchor's SEC-007, which is the rule about prohibiting dangerous subprocess calls. This means when a developer sees a violation, they see both the Anchor rule ID and the corresponding industry standard ID, so they immediately understand the regulatory context.

The third tier is government regulators. Anchor has rule packs for the Reserve Bank of India's FREE-AI framework, the EU AI Act of 2024, the SEC's 2026 examination priorities, SEBI's AI/ML consultation requirements, the CFPB's Regulation B guidance, and the FCA's AI governance standards. These regulator packs are opt-in. A bank in India activates the RBI pack. A broker-dealer in New York activates the SEC pack. A fintech operating in Europe activates the EU AI Act pack. The beauty is that the underlying violations are the same — only the regulatory lens changes.

Every single rule in the system has an ID like SEC-007 or ETH-002, a severity level (info, warning, error, or blocker), a minimum severity floor that cannot be lowered, a description, a detection method, and a mitigation suggestion. The severity ordering is strict: info is 0, warning is 1, error is 2, blocker is 3. This ordering matters because of how policy overrides work, which I will explain next.

## The Policy File: How Companies Customize Rules

Every company that uses Anchor gets their own private policy file called policy.anchor. This lives in a .anchor directory inside their project. The policy file lets the company do two things: raise the severity of existing rules, and add entirely new company-specific rules.

The first thing, raising severity, works like this. Suppose the constitutional rule SEC-006 is set to severity "error." A bank's compliance team decides that for their PCI-DSS scope, this should be a blocker — meaning it should prevent deployment entirely, not just flag a warning. They add an override in their policy.anchor that sets SEC-006 to "blocker." The loader reads this override and validates it against two constraints. First, the new severity must be greater than or equal to the existing severity. You can raise SEC-006 from error to blocker, but you cannot lower it from error to warning. The code calls a function called severity_gte that compares the numeric ordering. If you try to lower it, the system throws a ValueError with a message that says "The floor is absolute. policy.anchor can only raise severity." Second, the new severity must be at or above the rule's min_severity floor, which is set in the constitutional domain file and cannot be changed by the policy file at all.

This is the most important design decision in the entire system. The constitutional floor is immutable. No company, no developer, no administrator can weaken a governance rule. They can only make rules stricter. This is what makes Anchor trustworthy to regulators. When the RBI asks "can a bank disable your security rules?", the answer is provably no.

The second thing the policy file supports is custom rules. A company can add rules with the INTERNAL- prefix. For example, INTERNAL-001 might be a rule that detects unauthorized access to the company's internal vault API. These custom rules use the exact same detection engine as constitutional rules — they can use AST matching or regex patterns. They are scanned alongside the constitutional rules with no difference in treatment. The only constraint is that the prefix must match what the constitution declares, which by default is "INTERNAL."

The policy file also supports a third feature: custom AI provider registration. If a company uses a private AI provider that Anchor does not know about — say, an internal model hosted at api.internal-ai.company.com — they can declare it in the policy file under a custom_providers section. When the runtime interceptor starts, it reads these entries and adds them to the provider registry so that HTTP calls to those endpoints are also governed.

## Layer 1: The Static Compliance Engine

Layer 1 is the pre-deployment scanner. It runs before code reaches production. The developer runs a command called "anchor check" pointed at their repository, and the engine scans every source file against the full constitutional rule set.

The scanning process begins with the PolicyEngine class in engine.py. The engine receives two inputs: a list of rules (compiled from the constitution, frameworks, regulators, and the private policy file), and a list of file paths to scan. For each file, it first determines the programming language by looking at the file extension. A .py file goes to the PythonAdapter. A .ts or .tsx file goes to the TypescriptAdapter. There are five adapters total: Python, TypeScript, Rust, Go, and Java.

Each adapter implements a base class called LanguageAdapter that defines three abstract methods: build_dangerous_call_query, build_import_query, and build_inheritance_query. These methods take a list of target names (function names, module names, or class names) and return a tree-sitter S-expression query. An S-expression is a pattern-matching language that operates directly on the abstract syntax tree. For example, when the Python adapter needs to detect calls to subprocess.run, it generates this S-expression: a "call" node whose "function" child is either an "identifier" or an "attribute" node that matches the regex "subprocess\.run". The tree-sitter parser then walks the entire AST of the file and returns every node that matches this pattern.

The adapter calls get_grammar() to load the compiled tree-sitter grammar for its language. For Python, this imports tree_sitter_python and wraps it in a Language object. The adapter then creates a Parser from this grammar and parses the source file bytes into an AST. This parsing is fast — tree-sitter was designed for real-time editor parsing, so it handles multi-thousand-line files in milliseconds.

Once the AST is parsed, the engine iterates over every active rule. Each rule has a detection method, which is one of two modes. Mode A is Smart AST matching. If the rule has a "match" field in its YAML definition, the engine uses tree-sitter queries. The match field specifies a type (function_call, import, or inheritance) and the target name. The engine asks the adapter to build the appropriate S-expression, executes it against the AST, and then performs post-match verification. For function calls, it checks that the matched function name exactly matches the expected name. For imports, it checks the module name. For inheritance, it checks the parent class name. This two-step process (broad query + exact verification) eliminates false positives.

Mode B is regex fallback. If the rule has a "pattern" field instead of a "match" field, the engine falls through to simple line-by-line regex scanning. It splits the file into lines and runs re.search on each line against the pattern. This is less precise than AST matching but works for patterns that span string content or comments rather than code structure.

There is a critical subtlety in how tree-sitter queries are executed. The tree-sitter Python bindings have changed their API surface four times across versions 0.20 through 0.23. What was query.matches(node) in 0.20 became QueryCursor(query).matches(node) in some 0.22 builds, and QueryCursor().matches(query, node) in others. The engine handles this with what the code calls the "HYDRA Execution Strategy." It tries four different execution patterns in sequence: first the standard matches API, then QueryCursor with query-bound construction, then QueryCursor with separate arguments, and finally a captures() fallback. Whichever pattern succeeds on the developer's installed version is the one that runs. This makes Anchor work across all tree-sitter versions without requiring developers to pin a specific version.

When a violation is found, the engine checks for inline suppressions. A developer can write "# anchor: ignore SEC-007" at the end of a line to suppress a specific rule for that specific line. The engine detects this by checking the current line and the next line for the suppression comment. If found, it does not count the violation but instead records it separately as a suppressed finding. Critically, the engine also runs git blame on the suppressed line to record who authorized the suppression. This means the audit trail shows not just that a rule was suppressed, but which specific developer did it and when. The constitution mandates suppress_tracking: true, meaning all suppressions appear in the final report. You cannot silently ignore a rule — there is always a paper trail.

After all matches are processed, the engine performs ID aggregation. If a violation triggers SEC-007, the engine searches all loaded rules to find any rule where maps_to references SEC-007. If FINOS-014 and OWASP-LLM-02 both map to SEC-007, the violation ID becomes "FINOS-014, OWASP-LLM-02, SEC-007" — a comma-separated list of all affected standards. This is powerful for compliance reporting because a single code violation can simultaneously demonstrate non-compliance across multiple regulatory frameworks.

The final output of Layer 1 is a structured report containing the violations list, the suppressed findings list, and metadata about the scan (project name, git commit, file counts, timing). This report can be output as JSON for CI pipeline consumption or as formatted markdown for human review.
